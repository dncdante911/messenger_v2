## 2. АУДИТ NODE.JS СЕРВЕРА

### 2.1 Архитектура

```
main.js
  ├── Express (REST API)
  │   ├── routes/auth.js           — регистрация, OTP
  │   ├── routes/messaging.js      — история сообщений
  │   ├── routes/private-chats/    — ЛС (6 файлов)
  │   ├── routes/groups/           — группы (10 файлов)
  │   ├── routes/channels/         — каналы (10 файлов)
  │   ├── routes/stories/          — истории
  │   ├── routes/bots/             — боты
  │   ├── routes/calls.js          — звонки
  │   ├── routes/signal.js         — E2EE ключи
  │   ├── routes/users.js          — пользователи
  │   └── ... (translator, notes, recordings, subscription)
  ├── Socket.IO 4.8.3
  │   ├── listeners/listeners.js   — основные события
  │   ├── listeners/calls-listener.js — WebRTC (52KB)
  │   ├── controllers/             — 320+ обработчиков
  │   └── @socket.io/redis-adapter — горизонтальное масштабирование
  ├── Sequelize ORM → MySQL2
  │   └── models/ (170+ таблиц)
  └── Redis (кеш + pub/sub + Socket.IO adapter)
```

### 2.2 Что работает хорошо

| Компонент | Оценка | Комментарий |
|-----------|--------|-------------|
| Socket.IO кластеризация | ✅ Хорошо | Redis-адаптер обеспечивает multi-process |
| WebRTC сигналинг | ✅ Хорошо | calls-listener.js (52KB) — полная реализация |
| Авторизация запросов | ✅ Хорошо | ApiKeyInterceptor + NodeAuthInterceptor |
| Rate limiting | ✅ Хорошо | 300 req/min global, 15 req/15min auth |
| Sanitize HTML | ✅ Хорошо | sanitize-html + striptags |
| Health check endpoint | ✅ Хорошо | /api/health с метриками |
| SMTP каскад | ✅ Хорошо | localhost → mail.sthost.pro fallback |
| Signal Protocol routes | ✅ Хорошо | Pre-keys, bundles, group keys |
| DB connection pool | ✅ OK | max=20, timeout=15s |

### 2.3 Критические проблемы

| Проблема | Риск | Решение |
|---------|------|---------|
| **OTP в памяти** | При рестарте PM2 все OTP теряются, пользователи не могут войти | Перенести в Redis с TTL 600s |
| **Нет FCM push при оффлайн** | Пользователь не получает уведомления если Socket закрыт | Интегрировать Firebase Admin SDK |
| **DB pool = 20** | При 20 воркерах PM2 × 2 сервера = 40 процессов × 20 соединений = 800 DB-соединений — перегрузка MySQL | Использовать PgBouncer/ProxySQL или уменьшить pool до 5 |
| **Нет индексов на wo_messages** | При миллионах сообщений запросы деградируют | Добавить индексы: (from_id, to_id, time), (to_id, time) |
| **ORM N+1 запросы** | Sequelize может генерировать N+1 на списках чатов | Аудит запросов + eager loading |
| **Нет JWT refresh tokens** | Истёкшие токены не обновляются без повторного логина | Implement refresh token rotation |
| **Файлы загружаются на сам сервер** | Multer пишет на локальный диск — нет CDN failover | S3-совместимое хранилище или Strapi CDN |
| **Нет graceful shutdown** | При перезапуске PM2 активные WebSocket соединения рвутся | Добавить SIGTERM handler + drain timeout |

### 2.4 Средние проблемы

| Проблема | Решение |
|---------|---------|
| Нет pagination в chat list (chats-list.js) | cursor-based pagination |
| Отсутствует compression middleware | `compression` npm пакет для gzip |
| Нет HTTP/2 | Настроить nginx с HTTP/2 + h2c |
| Нет мониторинга (метрики) | Prometheus + Grafana или PM2 Keymetrics |
| Нет structured logging | Winston/Pino с JSON логами |
| Нет circuit breaker | `opossum` для устойчивости к сбоям Redis/DB |

### 5.2 Ключевые архитектурные различия

**Telegram использует MTProto:**
- Бинарный протокол поверх любого транспорта (TCP, UDP, HTTP)
- Идеально работает при плохом интернете (UDP mode)
- Оффлайн-очередь встроена в протокол
- Шифрование транспортного уровня не нужно (протокол сам шифрует)

**Viber использует Kafka:**
- Kafka гарантирует at-least-once delivery
- Horizontal partitioning по user_id
- Cassandra — write-optimized NoSQL, идеально для миллиардов сообщений
- Нет единой точки отказа

**WorldMates использует Socket.IO:**
- Проще в разработке, но менее эффективен
- Нет встроенной гарантии доставки (нужно реализовывать самому)
- Redis-адаптер решает задачу multi-process, но не multi-server failover
- MySQL — реляционная БД, горизонтальное масштабирование сложнее

## 6. АНАЛИЗ ЖЕЛЕЗА И ЁМКОСТЬ

### 6.1 Конфигурация серверов

```
Сервер A & B (идентичные):
  CPU: 2× Intel Xeon E5-2680v4 (2 × 14 ядер = 28 физических / 56 потоков)
  RAM: 64GB DDR4 ECC
  Storage:
    - 12TB HDD (медиа-файлы, архивы)
    - 512GB SSD NVMe M.2 (БД, индексы)
  Network: 200 Mbps (апгрейд до 2 Gbps)

PM2: 20 воркеров × 2 сервера = 40 Node.js процессов
Redis: предположительно на одном из серверов
MySQL: предположительно на одном из серверов (SSD NVMe)
```

### 6.2 Расчёт ёмкости

#### CPU (Node.js — однопоточный, I/O-bound)
```
Один Node.js воркер при типичной нагрузке мессенджера:
  - ~500-1500 одновременных Socket.IO соединений
  - ~2000-5000 req/s (лёгкие запросы)

20 воркеров × 2 сервера = 40 воркеров:
  Оптимистичный расчёт: 40 × 1000 сокетов = 40 000 соединений
  Реалистичный расчёт: 40 × 600 сокетов  = 24 000 соединений

⚠️ НО: 20 воркеров на 28 физических ядер — оптимально
   E5-2680v4 имеет HT (56 логических) — можно поднять до 48-52 воркеров
   НО это создаст проблему с DB pool (см. ниже)
```

#### RAM
```
Node.js воркер: ~150-250MB RSS (при активных сокетах)
20 воркеров × 200MB = 4GB на сервер (Node.js)
MySQL + буферы InnoDB: ~16-24GB (рекомендуется)
Redis: ~2-4GB
OS + прочее: ~4GB
---
Итого: ~30-35GB из 64GB ✅ Достаточно с запасом
```

#### Network (200 Mbps текущий)
```
Одно текстовое сообщение: ~1-5 KB (Socket.IO event + metadata)
Один активный пользователь: ~5-20 KB/s (при активном чате)
Голосовой звонок (OPUS): ~32-64 Kbps на участника
Видеозвонок (VP8/H.264 720p): ~1-2 Mbps на участника

200 Mbps / 10 KB/s = ~20 000 одновременно активных пользователей (текст)
200 Mbps / 64 Kbps = ~3 000 одновременных голосовых звонков
200 Mbps / 1.5 Mbps = ~133 одновременных видеозвонков

При апгрейде до 2 Gbps:
  Текстовые: ~200 000 активных
  Голосовые: ~30 000 звонков
  Видео: ~1 333 видеозвонков
```

#### MySQL (512GB NVMe)
```
Текущий pool: 20 соединений × 40 процессов = 800 соединений (!)
Рекомендация MySQL max_connections: 500-1000 (при 64GB RAM)
---
⚠️ ПРОБЛЕМА: 800 соединений без ProxySQL могут исчерпать MySQL
Решение: ProxySQL перед MySQL, pool на воркер снизить до 3-5
```

#### Storage (12TB HDD + 512GB NVMe)
```
NVMe (512GB):
  - MySQL данные + индексы
  - Redis AOF/RDB
  - Размер одного сообщения в БД: ~500B
  - 512GB / 500B = ~1 миллиард сообщений до заполнения ✅

HDD (12TB):
  - Медиа-файлы пользователей
  - Средний медиафайл: ~5MB
  - 12TB / 5MB = ~2.4 миллиарда файлов ✅ огромный запас
```

### 6.3 Итоговая оценка ёмкости

| Метрика | Текущий (200 Mbps) | При 2 Gbps |
|---------|-------------------|-----------|
| **Одновременных пользователей (онлайн, активных)** | **15 000 – 20 000** | **150 000 – 200 000** |
| Голосовых звонков одновременно | ~3 000 | ~30 000 |
| Видеозвонков одновременно | ~100-133 | ~1 000-1 333 |
| Зарегистрированных пользователей (суммарно) | ~1-5 млн (без деградации) | ~10-50 млн |
| Socket.IO соединений стабильно | ~20 000 - 40 000 | ~40 000 (CPU лимит) |

**Ключевое ограничение сейчас: пропускная способность сети (200 Mbps), не CPU/RAM.**

## 7. ПРИОРИТЕТЫ УЛУЧШЕНИЙ

### 7.1 🔴 Критические (нужно сделать немедленно)

## ИТОГОВЫЙ ВЫВОД

### Что у вас уже есть и что впечатляет:
- **Signal Protocol E2EE** — лучше чем у Telegram (MTProto) и Viber — это реальное конкурентное преимущество
- **Полный стек**: ЛС, группы, каналы, истории, боты, звонки, стримы — покрытие ~80% функционала Telegram
- **Боты с inline, опросами, webhook, RSS** — полноценная экосистема
- **Облачный бэкап на 3 провайдера** — лучше чем у конкурентов
- **Железо с запасом**: 2 сервера × 28 ядер × 64GB RAM могут держать 15-20K одновременных пользователей уже сейчас

### Главные слабости:
1. **Нет FCM push** — критическая дыра для UX
2. **Нет Mini Apps** — огромная экосистема, которую конкуренты уже имеют
3. **Нет глобального поиска** — базовая функция
4. **Узкое место — сеть (200 Mbps)** — апгрейд до 2 Gbps даст 10× пользователей
5. **MySQL без шардирования** — при 5M+ пользователей начнёт проседать

### Позиция в конкуренции:
```
Телеграм:   ████████████████████ (100%)
Viber:      ██████████████░░░░░░ (70%)
WorldMates: █████████████░░░░░░░ (65%)  ← вы здесь
```

**При реализации критических и высокоприоритетных пунктов** — через 3-4 месяца можно выйти на 75-80% от функционала Viber и быть вполне конкурентоспособным региональным мессенджером. Mini Apps + глобальный поиск + FCM — это "билет" в догоняющие.
