## 2. СЕРВЕРНАЯ ИНФРАСТРУКТУРА И ФИЗИЧЕСКАЯ БЕЗОПАСНОСТЬ

### 2.1 Географическое расположение серверов

Все производственные серверы WallyMates физически расположены в **Украине, г. Днепр**. Данные пользователей хранятся и обрабатываются исключительно в рамках данной юрисдикции, за исключением случаев, прямо предусмотренных настоящей Политикой и применимым законодательством.

### 2.2 Физическая безопасность

Дата-центры, в которых размещена инфраструктура WallyMates, обеспечивают:
- многоуровневый контроль физического доступа (электронные пропуска, биометрическая идентификация);
- круглосуточную охрану и видеонаблюдение;
- системы бесперебойного электропитания (ИБП + дизельные генераторы);
- климатический контроль и системы охлаждения серверного оборудования;
- защиту от пожара, затопления и аварийных ситуаций;
- журналирование всего физического доступа персонала.

### 2.3 Сетевая безопасность инфраструктуры

- Межсетевые экраны (Firewall) корпоративного класса с политиками «белых списков»
- Системы обнаружения и предотвращения вторжений (IDS/IPS)
- Многоуровневая защита от DDoS-атак
- Сегментация сети по зонам безопасности (DMZ, прикладной уровень, уровень базы данных)
- VPN для удалённого технического доступа персонала
- Непрерывный мониторинг и аудит сетевого трафика
- Ограничение входящих подключений по принципу минимально необходимых привилегий

### 2.4 Компоненты серверного стека

Платформа WallyMates работает на следующем стеке:
- **Веб-сервер:** Nginx (reverse proxy) с SSL/TLS терминацией
- **Node.js сервер** (порт 449): обработка сообщений в реальном времени, WebSocket, WebRTC
- **База данных:** MySQL 8.0+ — основное хранилище
- **Кеш и сессии:** Redis — сессии, статусы онлайн, временные данные
- **Управление процессами:** PM2 (автозапуск, мониторинг)
- **Медиахранилище:** локальное файловое хранилище, защищённое от прямого доступа

## 3. ШИФРОВАНИЕ

### 3.1 Сквозное шифрование личных сообщений (E2EE)

WallyMates реализует **Протокол Signal** (Double Ratchet + X3DH) для сквозного шифрования (End-to-End Encryption, E2EE) всей личной переписки между пользователями:

**X3DH — Extended Triple Diffie-Hellman:**
Протокол обмена ключами, позволяющий двум пользователям установить защищённый сессионный ключ асинхронно — без необходимости одновременного присутствия онлайн. Используется при первом контакте между пользователями.

**Double Ratchet Algorithm:**
После установки сессии каждое сообщение шифруется уникальным ключом. Алгоритм обеспечивает:
- *Forward Secrecy (прямая секретность):* компрометация текущего ключа не раскрывает содержимое предыдущих сообщений;
- *Break-in Recovery (постдеконспирация):* после компрометации одного ключа последующие сообщения автоматически защищаются новыми ключами.

**Технические параметры:**
- Алгоритм шифрования: **AES-256-GCM** (Galois/Counter Mode)
- Версия шифра: Signal Cipher Version 3
- Кривая: Curve25519 для обмена ключами

**Ключи хранятся исключительно на устройствах пользователей.** Серверы WorldMates не имеют доступа к закрытым ключам и технически не могут расшифровать содержимое личных сообщений.

**Одноразовые предварительные ключи (One-Time Pre-Keys, OPK):**
Каждый аккаунт генерирует набор одноразовых ключей для защиты первого контакта. Использованные ключи не восстанавливаются. Система автоматически пополняет запас OPK при его снижении.

### 3.2 Шифрование групповых сообщений

Для групповых чатов применяется **Sender Key Distribution Protocol** (часть экосистемы Signal):
- единый сессионный ключ отправителя для каждой группы;
- безопасная рассылка ключей каждому участнику при добавлении в группу;
- автоматическое обновление ключей при изменении состава группы.

### 3.3 Шифрование медиафайлов

Все медиафайлы (изображения, видео, аудио, документы) шифруются перед загрузкой на сервер:

| Параметр | Значение |
|----------|----------|
| Алгоритм | AES-256-GCM |
| Вектор инициализации (IV) | 12 байт, уникальный для каждого файла |
| Тег аутентификации | 16 байт (защита от подделки) |
| Автоочистка временных файлов | 1 час после расшифровки на устройстве |

### 3.4 Шифрование при передаче данных

- **TLS 1.2 / TLS 1.3** для всех HTTPS-соединений
- **WSS (WebSocket Secure)** для всех соединений реального времени (Socket.IO)
- Все API-запросы осуществляются исключительно по защищённому каналу HTTPS
- Проверка SSL-сертификатов на стороне клиентского приложения

### 3.5 Шифрование данных на устройстве пользователя

- **Android EncryptedSharedPreferences** (BouncyCastle AES-256) — хранение JWT-токенов, ключей сессии и чувствительных настроек
- Локальная база данных (Room Database) с поддержкой шифрования хранилища
- Резервные копии ключей шифруются перед экспортом в облачное хранилище
- Уникальные ключи идентификации устройства хранятся в зашифрованном виде

## 4. АУТЕНТИФИКАЦИЯ И УПРАВЛЕНИЕ ДОСТУПОМ

### 4.1 Методы аутентификации пользователей

| Метод | Описание |
|-------|----------|
| Телефон + SMS OTP | Верификация номера телефона через одноразовый SMS-код при регистрации и входе (Twilio) |
| JWT (JSON Web Token) | Токены с ограниченным сроком действия для авторизации всех API-запросов |
| Автообновление токенов | Прозрачное обновление JWT без повторного ввода учётных данных |
| 2FA — TOTP | RFC 6238 — поддержка приложений Google Authenticator, Authy и совместимых |
| Биометрия | Отпечаток пальца / распознавание лица (Android BiometricPrompt API, уровень Strong) |
| PIN-блокировка | Дополнительная блокировка приложения PIN-кодом |

### 4.2 Управление токенами и сессиями

- JWT-токены хранятся в зашифрованном хранилище (EncryptedSharedPreferences)
- Автоматическое обновление токенов при истечении срока
- Автоматический выход при длительном бездействии пользователя
- Уведомление пользователя о входе с нового или незнакомого устройства
- Журнал активных сессий с возможностью удалённого завершения любой сессии
- Серверные сессии хранятся в Redis с автоматическим TTL (истечением срока)

### 4.3 Хранение паролей

- Алгоритм хеширования: **bcrypt** (bcryptjs 3.x) с адаптивной стоимостью
- Уникальная криптографически стойкая соль для каждого пользователя
- Пароли **никогда** не хранятся в открытом тексте или обратимо зашифрованном виде
- Безопасный процесс восстановления пароля через верифицированный номер телефона

### 4.4 Контроль доступа персонала

- Принцип минимально необходимых привилегий (Least Privilege Principle)
- Ролевое управление доступом (RBAC) к производственным системам
- Обязательная многофакторная аутентификация (MFA) для доступа к серверной инфраструктуре
- Журналирование всех административных и технических действий с временны́ми метками
- Периодический пересмотр прав доступа персонала

## 5. ЗАЩИТА ДАННЫХ ПОЛЬЗОВАТЕЛЕЙ

### 5.1 Принцип минимизации данных

Компания собирает только те данные, которые объективно необходимы для предоставления услуг WallyMates:

**Обязательные данные:**
- Номер телефона (для идентификации и SMS-верификации)
- Идентификатор устройства (для push-уведомлений)

**Данные профиля (предоставляются добровольно):**
- Имя пользователя, имя и фамилия
- Фотография профиля / аватар
- Биография / статус
- Адрес электронной почты (для восстановления аккаунта)

**Технические данные (собираются автоматически):**
- IP-адрес (геолокация страны/региона, обнаружение злоупотреблений)
- Временны́е метки событий (отправка/получение/прочтение)
- Версия ОС и приложения
- FCM-токен (для push-уведомлений Firebase)

### 5.2 Инструменты автоматического удаления данных

| Функция | Описание |
|---------|----------|
| Таймер автоудаления сообщений | 1 день / 1 неделя / 1 месяц — по выбору пользователя для каждого чата |
| View Once | Медиафайл удаляется сразу после первого просмотра получателем |
| Истории | Автоматически удаляются через 24 часа |
| Временные файлы на устройстве | Расшифрованные медиафайлы удаляются через 1 час |
| Очистка истории чата | Полная очистка по запросу пользователя |

### 5.3 Удаление аккаунта

При удалении аккаунта пользователем:
- Данные профиля удаляются немедленно
- Все сообщения пользователя удаляются с серверов
- JWT-токены и сессии аннулируются незамедлительно
- Оставшиеся данные в резервных копиях удаляются в течение **30 дней**
- Исключение: данные, которые по закону необходимо сохранить для расследования или судопроизводства

## 6. МОДЕРАЦИЯ КОНТЕНТА

### 6.1 Автоматическая система модерации

Для публичного контента (каналы, публичные группы) применяется многоуровневая автоматическая модерация:

**Уровень 1 — Машинное обучение (NudeNet):**
- Анализ изображений на предмет NSFW-контента
- Порог автоматической блокировки: 85%+ уверенности
- Применяется к публичным каналам при загрузке медиафайлов

**Уровень 2 — Хэш-база запрещённого контента:**
- SHA-256 хэши ранее удалённых файлов
- Перцептивные хэши (pHash) для обнаружения визуально схожего контента
- Мгновенная блокировка при совпадении с базой

**Уровень 3 — Ручная модерация:**
- Очередь спорного контента для проверки модераторами
- Рассмотрение жалоб пользователей
- Аудит решений автоматических систем

### 6.2 Политики контента по типам каналов

| Тип канала | Политика |
|------------|----------|
| Публичные каналы | Строгая — автоматическая блокировка NSFW-контента |
| Новостные каналы | Умеренная — размытие контента с предупреждением |
| Приватные группы | Базовая — блокировка абсолютно запрещённых категорий |

### 6.3 Категории жалоб

Пользователи могут сообщить о нарушениях:
- Спам / нежелательные рассылки
- Преследование / харассмент
- Угрозы насилием
- Запрещённый сексуальный контент (CSAM — немедленный приоритет)
- Пропаганда ненависти или дискриминации
- Терроризм / экстремизм
- Мошенничество
- Другие нарушения
